# Add resource_usage_logs table to SQLite

Closes #164

## What

Introduces a dedicated `resource_usage_logs` table that records per-transaction
CPU, memory, and fee-parameter snapshots for any tracked Soroban contract.

Previously, resource data was tied exclusively to TTL-extension transactions
in `extension_history`. This new table is transaction-agnostic, making it
usable by any future code path that wants to persist resource consumption
without performing an extension.

## Schema

New table added to `src/db/schema.sql`:

```sql
CREATE TABLE IF NOT EXISTS resource_usage_logs (
    id                       INTEGER PRIMARY KEY AUTOINCREMENT,
    contract_id              TEXT    NOT NULL REFERENCES contracts(id) ON DELETE CASCADE,
    cpu_insns                INTEGER NOT NULL,
    mem_bytes                INTEGER NOT NULL,
    fee_instructions         INTEGER,
    fee_read_ledger_entries  INTEGER,
    fee_write_ledger_entries INTEGER,
    fee_read_bytes           INTEGER,
    fee_write_bytes          INTEGER,
    fee_transaction_size     INTEGER,
    fee_historical_ledger    INTEGER,
    fee_rent_ledger          INTEGER,
    fee_refundable           INTEGER,
    recorded_at              TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_resource_usage_logs_contract_id ON resource_usage_logs(contract_id);
CREATE INDEX IF NOT EXISTS idx_resource_usage_logs_recorded_at  ON resource_usage_logs(recorded_at DESC);
```

All fee columns are nullable — not every RPC response includes a full fee breakdown.
`cpu_insns` and `mem_bytes` are required as they are always present in a simulation result.

A migration file (`src/db/migrations/001_resource_usage_logs.sql`) is also
included so that existing `sorokeep.db` files are upgraded automatically on
next startup via the `Migrator`.

## Repository API (`src/db/repositories.ts`)

Three new exported functions following the exact patterns used by the rest
of the file:

| Function | Description |
|---|---|
| `insertResourceUsageLog(db, log)` | Insert a new log entry; returns the new row id |
| `getResourceUsageLogs(db, contractId, options?)` | Query logs newest-first; supports `limit` and `since` filters |
| `getLatestResourceUsageLog(db, contractId)` | Convenience helper — returns the single most-recent row |

## Tests

Written **before** the implementation (TDD):

`tests/db/resource_usage_logs.test.ts` — 20 tests covering:

- Schema: table existence, index existence, foreign-key enforcement
- `insertResourceUsageLog`: happy path, null fee columns, duplicate entries
  allowed, invalid `contract_id` rejected, zero values, caller-supplied
  timestamp
- `getResourceUsageLogs`: empty result, DESC ordering, per-contract isolation,
  `limit` option, negative-limit guard, `since` filter
- `getLatestResourceUsageLog`: empty result, most-recent selection, cross-contract isolation
